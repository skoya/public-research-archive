Key Integration Patterns

ID Prefix Pattern (CDP v2)

Coinbase CDP uses typed prefixes on IDs: account_, entity_, customer_. The prefix encodes entity type in the ID itself, allowing callers to determine the type without an extra lookup. Prime uses bare UUIDs — entity type is inferred from API path context.

Polymorphic Ownership (CDP v2)

The owner field on Account encodes both the ID and type. entity_<uuid> = organisation; customer_<uuid> = individual customer. No join required to determine owner type — just parse the prefix.

Quote-then-Execute (CDP v2 Transfer)

Transfers are created in quoted state with a fee estimate and an expiry window (typically 10–15 min). Callers review the quote, then call /execute. Set execute: true at creation to skip the review step. Expired quotes must be recreated.

Wallet Type Selection (Prime)

TRADING: funds needed for active order flow. VAULT: long-term cold storage with withdrawal controls. ONCHAIN: DeFi / direct onchain settlement. QC: regulated custody arrangement. Balance availability differs by type — TRADING funds are immediately available; VAULT requires withdrawal workflow.

Balance Sub-buckets (Prime)

amount = gross total. Subtract holds for funds available to trade. withdrawable_amount = net moveable. Staking wallets add bonded, unbonding, and claimable_rewards buckets. Use fiat_amount for USD-equivalent portfolio valuation.

Metadata Pass-through (CDP v2)

Both Transfers and DepositDestinations accept up to 10 key-value metadata pairs (keys ≤40 chars, values ≤500 chars). Metadata persists through the full lifecycle and appears on webhooks — useful for carrying internal reference IDs without API changes.

Idempotency

CDP v2 Transfers accept an X-Idempotency-Key header (≤128 chars) to prevent duplicate processing on retry. Prime Orders accept a client_order_id field for the same purpose. Always set these on creation requests in retry loops.

DepositDestination Subtype Pattern

The parent object holds shared fields; type-specific fields live in a discriminated sub-object. type=crypto → network + address fields present. type=fiat → bank details present. This table-per-type pattern is extensible — future rails add new subtype fields without changing the parent schema.

Transfer & Order Mechanics

Standard withdrawals, conversions, advanced order types, approval gates, fees, and Travel Rule conditions

Standard Withdrawal — POST /v1/portfolios/{portfolio_id}/wallets/{wallet_id}/withdrawals

destination_type options DESTINATION_BLOCKCHAIN — send to an external on-chain address. Requires blockchain_address.address + optional account_identifier (memo/tag) + network.
DESTINATION_PAYMENT_METHOD — fiat withdrawal to a linked bank (ACH, wire, SEPA). Requires payment_method.payment_method_id.
DESTINATION_WALLET — internal transfer to another Prime wallet (no on-chain settlement).
DESTINATION_COUNTERPARTY — send to a pre-registered counterparty. Requires counterparty.counterparty_id.
Required fields alwaysamount (whole units string), currency_symbol, destination_type, idempotency_key
Responseactivity_id, approval_url, symbol, amount, fee (network fee), destination_type, source_type, transaction_id
Approval gateWithdrawals generate an Activity. Many withdrawal types require consensus approval in the Prime UI before funds move. The approval_url in the response links directly to the approval screen. Poll the activity status until COMPLETED.
FeesNetwork fee (fee field) assessed at submission for blockchain withdrawals. Fiat withdrawals may carry rail-specific fees. Fee is informational in the response — deducted from wallet balance at settlement.

Conversion (Swap) — POST /v1/portfolios/{portfolio_id}/wallets/{wallet_id}/conversion

Supported pairsUSD ↔ USDC and USD ↔ PYUSD only. Source wallet must hold the source_symbol asset.
Required fieldsamount (whole units), destination (destination wallet UUID), source_symbol, destination_symbol, idempotency_key
Responseactivity_id, source, destination, source_symbol, destination_symbol, amount, transaction_id
Settlement1:1 rate, settled internally. No on-chain transaction required — it is a ledger debit/credit between two wallets within the same portfolio.

Order Types — POST /v1/portfolios/{portfolio_id}/order

Required for all orders: client_order_id · product_id · side (BUY|SELL) · type · base_quantity OR quote_value
Type Additional required fields Behaviour
MARKETnoneExecutes immediately at best available price. No price guarantee. Cannot be cancelled once submitted.
LIMITlimit_priceRests on order book until filled at or better than limit_price. Supports post_only=true to reject if it would immediately match (maker-only). Supports GTC / GTD / IOC / FOK time_in_force.
STOP_LIMITstop_price, limit_priceInactive until market reaches stop_price, then becomes a limit order at limit_price.
TWAP limit_price, start_time, expiry_time Advanced. Time-Weighted Average Price. Splits order into equal time-sliced child orders over the interval [start_time, expiry_time]. Min notional $100/hr. Targets average market price over the window. Use to reduce market impact for large orders.
VWAP limit_price, start_time, expiry_time Advanced. Volume-Weighted Average Price. Slices order in proportion to historical volume distribution over the window. Min notional $100/hr. Benchmarks against the period VWAP.
BLOCK Negotiated off-book Advanced. Large off-exchange block trades negotiated bilaterally. Settled as a single fill. Avoids market impact of the lit order book. Suitable for institutional-size positions.
RFQ Counterparty negotiation Advanced. Request for Quote. Counterparties provide firm quotes; client accepts or rejects before expiry. Execution at agreed price, not market price. Common for illiquid pairs or large notional.
PEG peg_offset_type, offset, wig_level Advanced. Pegged order that dynamically tracks the best bid/ask with a configured offset. Continuously reprices. wig_level controls aggressiveness when top-of-book changes. Useful for passive participation strategies.
Iceberg orders: Use display_base_size / display_quote_size on LIMIT orders to show only a portion on the order book; each slice refreshes as it fills.  |  Self-trade prevention: set stp_id to prevent your own orders crossing each other.

Entity Destruction & Deletion Conditions

What can be deleted, what preconditions apply, and what happens to child entities

Entity Deletable? Preconditions Child / side-effect behaviour
ORGANIZATION No self-serve Offboarded by Coinbase only. All sub-entities, portfolios, and wallets must be zeroed and closed first. Entire hierarchy (Entities, Portfolios, Wallets) archived or purged by Coinbase operations team.
ENTITY No self-serve Contact Coinbase. All portfolios under the Entity must have zero balances and be closed first. Portfolios and their wallets become inaccessible when the parent Entity is closed.
PORTFOLIO Limited All wallet balances must be zero. All open orders must be cancelled or filled. No pending Activities. Initiated via Prime UI or account management — no public API delete endpoint. All child Wallets deactivated. Historical Activity, Order, and Transaction records retained for audit. Deposit addresses become invalid.
WALLET (TRADING / QC) Not deletable Auto-created system wallets (one per asset per portfolio). Cannot be individually deleted — deactivated only when the parent Portfolio is closed. Balances must be withdrawn before portfolio closure. Historical balance snapshots retained.
WALLET (VAULT) Via UI only Balance must be zero (withdraw or transfer all assets first). No pending withdrawal Activities. Closure requires consensus approval in Prime UI. Wallet ID becomes inactive. Deposit addresses linked to the vault are permanently invalidated — never reuse them. Historical transactions retained.
WALLET (ONCHAIN) Via UI only Balance must be zero. No pending on-chain transactions. Key decommission handled by Coinbase custody. Consensus approval required. The blockchain address is permanently retired. Any future sends to that address will not be credited. Wallet ID deactivated.
ORDER Cancellable Must be in OPEN state. MARKET orders cannot be cancelled once submitted. Partial fills cancel the remaining open quantity only. Filled portions are irreversible. Cancelled orders retain their record with status CANCELLED. Commission on filled portions is non-refundable.
ACTIVITY Not deletable Immutable audit records. Can be approved or rejected while pending, but never deleted. Rejection of a pending Activity cancels the underlying operation (vault creation, withdrawal) but the Activity record is permanently retained.
STAKING_POSITION Unstake only Cannot be instantly removed. Must call Unstake API. ETH: enters UNBONDING; validator queue exit time varies hours to days; partial unstake supported (min 1 ETH); rewards continue during unbonding. SOL: full unstake only; activates at epoch boundary; 2–4 day unbonding. On COMPLETED, staked balance returns to wallet as withdrawable. Historical staking and reward records retained permanently.
DEPOSIT_ADDRESS Not deletable Cannot be deleted or rotated individually. Invalidated only when the parent Wallet is decommissioned. Warning: unique-type addresses are generated per-payment and should not be reused. Static addresses persist as long as the wallet is active.
Key irreversibility rules:
  • Completed / settled transactions (fills, withdrawals, conversions) are permanent and non-refundable.
  • Decommissioned blockchain addresses must never be reused — funds sent to an inactive address may be unrecoverable.
  • Activity records are immutable audit logs — rejection changes the outcome but not the record.
  • VAULT wallet closures and ONCHAIN wallet decommissions require consensus approval and cannot be expedited.
  • ETH validator exit queue is protocol-determined — there is no API override to accelerate unbonding.
